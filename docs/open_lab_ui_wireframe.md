# 公开实验室 (Open Lab) UI/UX 交互线稿与视觉规划 (基于 Idea -> Feature 架构)

## 1. 整体视觉风格 (Visual Tone)
*   **风格基调**：**Brutalism (粗野主义) 混搭 Geek 极简风**。大字号、粗线条、高对比度的黑白灰主色，配合一个极其跳跃的“品牌色”（比如荧光绿/电光蓝）作为唯一的视觉焦点（如点赞按钮、状态标签）。
*   **排版逻辑**：打破常规的对称网格，采用“卡片瀑布流”的排版。外层是“大灵感(Ideas)”，内层是“具体功能(Features)与进度”。

---

## 2. 页面结构线稿 (Wireframe)

[ 小字描述 ] 在这里，每一个痛点都可能变成下一个产品。提出你的想法，决定我的代码。

### 2.3 灵感详情页 (Idea Detail / Modal) - 核心双轨互动区
当用户点击某一个 Idea 后进入的详情页面。采用**左右分栏**（PC端）或**上下分栏**（移动端）。

**情景 A：已立项/研发中的 Idea (如：订阅监控工具)**
```text
[ 返回看板 ]   [ 分享此灵感 ]

================== 顶部：核心概念区 ==================
[ 独立开发者订阅收入大盘监控工具 ]
提出者：@Alex | 状态：🚧 研发中
[ 👍 我也需要这个产品 (128票) ] (超大按钮，点击弹跳特效)

----------------------------------------------------
[ 左栏/主栏：Builder 的造物日志 (Changelog / Timeline) ]

👩‍💻 @JESS.PU (Builder)：
> 2月28日：产品初步 UI 设计稿出炉。
> [大图展示]

👩‍💻 @JESS.PU (Builder)：
> 2月27日：立项！决定先跑通 Stripe 和 支付宝的接口。

(像博客的时间线，按时间倒序排布，展现产品的生命演进)

----------------------------------------------------
[ 右栏/侧边栏：子需求/功能共创区 (Feature Requests) ]

[ 标题：你希望这个产品包含什么功能？ ]
[ 输入框：+ 提议一个具体功能... ]

> 功能提议 1：希望能支持微信支付的流水接入 (👍 12票)
  └ @Builder 回复：这个做进 v1.1 里，微信接口正在申请。
----------------------------------------------------
```

**情景 B：未立项/等待共鸣的 Idea (如：微信读书插件)**
```text
[ 返回看板 ]   [ 分享此灵感 ]

================== 顶部：核心概念区 ==================
[ 微信读书笔记一键转 Notion 插件 ]
提出者：@某网友 | 状态：💡 等待共鸣
[ 👍 猛击这里催更 (45票) ] (此时文案偏向“催更/投票”)

----------------------------------------------------
[ 左栏/主栏：Builder 的初步评估 (Initial Thoughts) ]

(此时尚未立项，这里展示的是“立项条件”或“Builder 评估”)

👩‍💻 @JESS.PU (Builder)：
> 【初步评估】这个痛点很痛！不过目前还需要明确一下：大家是更想要一个纯本地的桌面端，还是一个浏览器插件？请在右侧留言，满 100 票我立刻开始画原型！

(如果连评估都没有，则展示幽默的占位图：比如一张咖啡杯底朝天的图，配字：“Builder 还在观察这个灵感的潜力，快用投票砸醒他！”)

----------------------------------------------------
[ 右栏/侧边栏：子需求/功能共创区 (Feature Requests) ]

[ 标题：如果这个做出来，你最想要什么功能？ ]
[ 输入框：+ 提议一个具体功能... ]

> 功能提议 1：必须是浏览器插件才方便！(👍 20票)
> 功能提议 2：希望能自动带上原文的高亮样式 (👍 15票)
----------------------------------------------------
```
*   **交互亮点**：
    *   **双轨并行**：左边是 Builder 的主场（秀进度、秀肌肉、讲故事）；右边是用户的客场（提功能、投票、激烈讨论）。
    *   **颗粒度清晰**：用户一开始在外部只对“大概念”投票，进来以后才会对“细枝末节（比如有没有深色模式、接不接特定支付）”进行提议。这就解决了你说的“Idea vs 软件具体功能”的矛盾。

---

## 3. 核心设计原则总结 (Design Principles)
1. **层次分明**：看板层只展示“宏大叙事（Idea）”，详情层才展示“具体功能（Features）”。
2. **Builder 的舞台**：左侧/主干的进度流是视觉核心，强化“有人正在为你写代码”的真实感。
3. **微共创体验**：把右侧的功能区做成类似 GitHub Issues 列表的感觉，但是更轻量、更易点赞（Reddit 的 Upvote 体验）。
