# 订阅与用户体系 (Subscribe & Auth) 产品需求与交互说明

## 1. 核心理念与定位
*   **极致的低阻力 (Frictionless)**：在这个注意力稀缺的时代，让用户设置密码是巨大的转化阻碍。本站彻底抛弃传统的账号密码体系。
*   **邮箱即身份 (Email is Identity)**：用户只需要输入一个邮箱，即可完成“订阅、身份绑定、后续互动”的全部闭环。
*   **“三合一”转化收口**：`/subscribe` 页面不只是一个表单，而是站长私域沉淀的总枢纽，承载“邮件订阅”、“直接打赏”和“添加微信”三大功能。

---

## 2. 无密码用户体系 (Passwordless Auth System)

### 2.1 身份认证逻辑 (Magic Link / OTP)
*   **首次互动拦截（如在 Open Lab 留言 / 提交 Idea 时）**：
    *   用户未登录状态下点击“提交”，弹出免密认证弹窗。
    *   用户输入邮箱，系统**必须向该邮箱发送一个 6 位验证码 (OTP) 或免密登录链接 (Magic Link)**。
    *   *防滥用机制*：用户必须填入验证码，或点击邮件里的链接后，其“留言/提交”动作才会真正生效，并且在设备端种下长期的 JWT/Cookie 登录态。这就杜绝了用户随便填一个别人的邮箱去恶意注册或发言的问题。
*   **状态保持**：一旦验证通过，保持长久登录态（比如设置过期时间为 90 天）。用户下次再来，直接可以点赞和以真实身份留言。

### 2.2 CMS 底层支撑 (`Users` 集合扩展)
*   `email`: 必填，唯一键。必须经过 OTP/Magic Link 验证。
*   `nickname`: 选填，前端生成默认的（如 Guest_1234），用户后续可改。
*   `avatarUrl`: 选填，默认可以用邮箱的 Gravatar 头像。
*   `isSubscribed`: Boolean，表示是否勾选了“接收邮件通知”。
*   `role`: `admin` | `user`。站长的账号设为 `admin`，享有特殊 UI 徽章。

## 3. 订阅页设计 (`/subscribe`)

### 3.1 页面结构 (Bento Box / Card 风格)
摒弃老套的长列表，采用几个精致的卡片（Card）并行展示，给用户“自助点餐”的感觉。

```text
[ 页面主标题 ] 保持连接
[ 页面副标题 ] 没有密码，没有废话。选择你喜欢的方式和我交流。

================== 转化卡片区 (并排或网格排列) ==================

[ 卡片 1：邮件通讯 (Newsletter) ] (最核心，面积最大)
- 标题：我的内参周报
- 描述：每月一到两封，只发我跑通的极客工作流、最新的商业观察，以及不对外公开的失败复盘。
- 状态判断：
  - 【未登录状态】：显示 [ 输入你的邮箱地址... ] [ 发送验证码登录并订阅 ]
  - 【已登录状态】：显示当前用户头像与邮箱，展示一个简单的 Switch 开关：[ 开启内参周报订阅 ]。

[ 卡片 2：即时通讯 (WeChat) ]
- 标题：加个微信
- 描述：适合商业合作、直接探讨，或者只是想找个同行聊聊天。
- 视觉：一张设计优雅的微信二维码 (QR Code)。
- 提示小字：“请备注来意（比如：来自博客）”。

[ 卡片 3：直接打赏 (Buy me a coffee) ]
- 标题：为创造力买单
- 描述：如果这个网站的内容帮到了你，甚至帮你省了钱，欢迎请我喝杯咖啡，支持服务器运转。
- 操作：[ 微信支付按钮 ] [ 支付宝按钮 ] (点击展开相应的打赏码)。
- (进阶版): 在这里罗列一个“光荣榜 (Sponsors)”，拉取所有赞助过的人的昵称。

======================================================================
```

### 3.2 交互闭环与通知 (Notifications)
*   **订阅成功动效**：当用户在卡片 1 输入邮箱点击订阅后，输入框原地变成一个大大的 ✅，并附文“欢迎加入”。
*   **站长感知通道**：通过 Webhook (如 Zapier/Make 或自定义 Node 脚本)，当有新用户输入邮箱订阅或打赏时，自动向站长的微信/飞书/企业微信推送一条消息通知：“🎉 新的同路人加入了：xxx@gmail.com”。
